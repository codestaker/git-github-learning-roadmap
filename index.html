üìå Git & GitHub Learning Roadmap

A step-by-step guide to mastering Git and GitHub, from beginner to advanced. We'll build a real project called MyPortfolio along the way.

üî∞ Beginner Level

1. Introduction to Git & GitHub

Before we start building our portfolio, let's understand our tools.

  • What is Git? It‚Äôs a Version Control System (VCS). Think of it as a "save" button for your code that lets you track every change and go back in time if you make a mistake.
  • What is GitHub? It‚Äôs a website to host your Git projects. It's like a social network for coders where you can store your code, collaborate with others, and showcase your work.

2. Setting up Your Environment

First, you need to install and configure Git on your computer.

For Windows:

  1. Download the installer from the official Git for Windows website.
  2. Run the installer and follow the setup instructions. The default options are usually fine for beginners.
  3. Once installed, you can use the "Git Bash" application that comes with it, which provides a powerful command-line experience.

For macOS & Linux:

# Install on Mac (using Homebrew)
brew install git

# Install on Linux (Debian/Ubuntu)
sudo apt install git

After installing, open your terminal (or Git Bash on Windows) and check the version to confirm it's installed correctly:

# Check the version
git --version

Now, introduce yourself to Git. This name and email will be attached to every commit you make.

git config --global user.name "Your Name"
git config --global user.email "your@email.com"

3. Creating Your First Project: MyPortfolio

Let's create a folder for our project and initialize Git inside it.

mkdir MyPortfolio
cd MyPortfolio
git init

The git init command creates a hidden .git folder, which is the "brain" of your repository where Git stores all the version history.

4. The Basic Workflow: Add & Commit

Let's create a simple homepage for our portfolio and save our first snapshot.

echo "
Welcome to My Portfolio
" > index.html
git status
git add index.html
git commit -m "Create initial homepage"
  • git status: Shows you the current state of your project (what's changed, what's new).
  • git add : Stages a specific file, preparing it for the next commit.
  • git add . : Stages all new and modified files in the current directory. This is a useful shortcut when you want to commit everything you've worked on.
  • git commit: Saves a snapshot of the staged files to your project's history. The -m flag is for the commit message.

5. Viewing Your Project's History

You can see the list of commits you've made.

git log
git log --oneline

--oneline gives you a much more compact view of the history.

6. Connecting to GitHub

It's time to put your project on GitHub!

  1. Go to GitHub.com and create a new repository named MyPortfolio.
  2. Connect your local project to this new GitHub repository and push your code.
 git branch -M main
                        git remote add origin https://github.com/YOUR_USERNAME/MyPortfolio.git

git push -u origin main

Now, refresh your GitHub page. You should see your index.html file!

7. Cloning a Repository

Cloning is how you get a copy of a Git repository from a remote source like GitHub. For example, to download your portfolio project on another computer:

git clone https://github.com/YOUR_USERNAME/MyPortfolio.git

8. Ignoring Files with .gitignore

Some files (like environment variables or dependency folders) shouldn't be saved in your repository. We use a .gitignore file to tell Git to ignore them.

echo "node_modules/" > .gitignore
echo ".env" >> .gitignore

Don't forget to commit this file!

git add .gitignore
git commit -m "Add .gitignore to exclude files"

‚ö° Intermediate Level

1. Branching: Working on New Features

Branches are one of the most powerful features in Git. They allow you to create an independent line of development to work on new features or bug fixes without affecting the main codebase.

Why create a branch?

  • Safety: It provides a safe, isolated space for your work. If you make a mistake, you won't break the stable `main` branch that your team relies on.
  • Collaboration: Multiple people can work on different features in parallel, each on their own branch.
  • Organization: It keeps your commit history clean and organized. Each feature's history is contained within its own branch until it's ready to be merged.

Let's create a branch for the new contact page in our portfolio project.

# Create a new branch called 'feature-contact'
git branch feature-contact

# Switch to the new branch to start working on it
git checkout feature-contact

Now you're on the new branch. Any commits you make here will not affect the `main` branch. Let's add a contact page.

echo "
Contact me at my@email.com
" > contact.html
git add contact.html
git commit -m "Add contact page"

2. Merging: Combining Your Work

Once your feature is complete and tested on its branch, you can merge it back into your main branch to integrate it into the main project.

# First, switch back to the branch you want to merge INTO
git checkout main

# Now, merge the feature branch INTO main
git merge feature-contact

Your `main` branch now contains all the work from the `feature-contact` branch, including the new `contact.html` file. It's now safe to delete the feature branch if you no longer need it.

# Delete the local feature branch
git branch -d feature-contact

3. Resolving Merge Conflicts

Sometimes, Git can't automatically merge changes if the same lines were edited on both branches. This is a merge conflict. You have to manually edit the file to resolve the conflict, then commit the fix.

# After manually fixing the file
git add conflicted_file.html
git commit -m "Resolve merge conflict"

4. Working with Remotes: Push, Pull, Fetch

To keep your local project and GitHub in sync:

  • git push: Sends your committed changes to the remote repository.
  • git pull: Fetches changes from the remote repository and merges them into your current branch.
  • git fetch: Fetches changes from the remote repository but doesn't merge them. This is useful to see what others have done before merging.
# Push your new contact page feature branch to GitHub
git push origin feature-contact

# Update your local main branch with changes from GitHub
git pull origin main

Common Error: Push Rejected (Fetch First)

Sometimes, when you try to push, you might get an error like ! [rejected] main -> main (fetch first). This is a very common and normal situation!

It simply means that there are changes on the remote repository (on GitHub) that you don't have on your local machine yet. Git is preventing you from accidentally overwriting those changes.

The Solution: You need to pull the remote changes, merge them with your local work, and then you'll be able to push.

# Pull the latest changes from the remote main branch
git pull origin main

# Git will merge the remote changes into your local files.
# If there are no conflicts, you can now push your work.
git push origin main

5. Collaboration: Forks & Pull Requests (PRs)

This is the heart of collaboration on GitHub. To contribute to someone else's project:

  1. Fork the repository on GitHub (creates your own server-side copy).
  2. Clone your fork to your local machine.
  3. Create a new branch for your changes.
  4. Make your changes, commit, and push to your fork.
  5. Open a Pull Request on GitHub to propose your changes to the original project.

6. Undoing Mistakes

Everyone makes mistakes. Here's how to fix them:

# Undo the last commit, but keep your changes
git reset --soft HEAD~1

# COMPLETELY undo the last commit and delete the changes
git reset --hard HEAD~1

# Revert changes to a file that haven't been committed yet
git checkout -- file.html

7. Tagging & Releases

When your portfolio is ready for its first official version, you can tag it.

git tag v1.0
git push origin v1.0

On GitHub, you can then create a "Release" from this tag, adding release notes.

8. Stashing Changes

Need to quickly switch branches but your work isn't ready for a commit? Use `git stash`.

# Temporarily save your uncommitted changes
git stash save "WIP - new navbar"

# To get your changes back
git stash pop

Troubleshooting Common Git Errors

Here are some common errors you might encounter and how to solve them.

1. `fatal: not a git repository`

Problem: You're trying to run a Git command in a folder that isn't a Git repository.

Solution: You either need to navigate to the correct folder, or initialize a new repository in the current folder with `git init`.

# Navigate to your project folder
cd /path/to/MyPortfolio

# Or initialize a new repository
git init

2. `error: Your local changes would be overwritten by merge`

Problem: You have uncommitted local changes, and you're trying to `git pull` changes that would conflict with them.

Solution: You must commit your local changes or stash them before you can pull.

# Option 1: Commit your changes
git add .
git commit -m "My local work"
git pull origin main

# Option 2: Stash your changes
git stash
git pull origin main
git stash pop

3. `fatal: remote origin already exists`

Problem: You're trying to add a remote named `origin` using `git remote add`, but one with that name already exists.

Solution: You can either update the existing remote's URL or remove it and add the new one.

# To see your existing remotes
git remote -v

# To change the URL of the existing origin
git remote set-url origin https://github.com/NEW_USERNAME/NEW_REPO.git

4. `fatal: refusing to merge unrelated histories`

Problem: This often happens when you create a remote repository with a file (like a README) and then try to pull it into a local repository that has its own, separate commit history.

Solution: You can allow Git to merge the two different histories with a special flag. Use this with care.

git pull origin main --allow-unrelated-histories

5. "Detached HEAD" State

Problem: This sounds scary, but it just means you've checked out a specific commit instead of a branch. You're no longer on a branch, so any new commits you make won't belong to any branch.

Solution: The easiest way to fix this is to check out the branch you want to be on.

# Switch back to the main branch
git checkout main

# If you made commits in the detached state and want to save them:
# Create a new branch from your current (detached) state
git branch temp-work
git checkout main
git merge temp-work

6. Overwriting Remote History with `git push --force`

Problem: You've tried `git pull`, but you still can't push, or you've decided that your local version of the project is the correct one and you want to completely overwrite what's on the remote repository.

Solution: You can use a "force push". WARNING: This is a destructive command. It will permanently delete any history on the remote branch that you don't have locally. Only use this if you are absolutely sure you won't be overwriting someone else's work.

# This will overwrite the remote 'main' branch with your local 'main' branch
git push --force origin main

üöÄ Advanced Level

1. Rebasing: A Cleaner History

git rebase is an alternative to `merge`. It rewrites commit history to create a straight, linear history. It can make your project history much easier to read.

git checkout feature-about
git rebase main

Warning: Don't rebase branches that are shared with other people.

2. Interactive Rebase

A powerful tool to clean up your commit history before merging. You can reorder, squash (combine), or reword commits.

git rebase -i HEAD~3

3. Cherry-Picking

Lets you pick a single commit from one branch and apply it to another. Useful for grabbing a quick bug fix from another branch.

git checkout main
git cherry-pick

4. GitHub Actions (CI/CD)

Automate your workflow. For our portfolio, you could set up a GitHub Action to automatically test your HTML for errors or deploy it to a web server every time you push to the `main` branch.

5. Git Hooks

Scripts that run automatically at certain points in the Git workflow. For example, a `pre-commit` hook could run a linter on your code before you're allowed to commit.

# Example: A simple pre-commit hook
echo 'echo "Running pre-commit checks..."' > .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit

6. Git Bisect: Finding Bugs

An incredibly useful tool for finding exactly which commit introduced a bug. It performs a binary search through your commit history.

git bisect start
git bisect bad      # Mark the current commit as broken
git bisect good  # Mark a past commit as working

Here's how you can apply everything you've learned to the MyPortfolio project:

  1. Initialize the project locally with `git init`.
  2. Create the `index.html` homepage, commit it, and push to a new repository on GitHub.
  3. Create feature branches for new pages: `feature-about` and `feature-contact`.
  4. On each branch, create the new HTML file, commit it, and push the branch to GitHub.
  5. Open Pull Requests on GitHub to merge your feature branches into `main`.
  6. Add a `.gitignore` file to ignore files like `.env`.
  7. Once you're happy with the site, tag a `v1.0` release.
  8. Use GitHub Pages to deploy your portfolio to the web for free!
